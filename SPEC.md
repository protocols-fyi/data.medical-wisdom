# Consumer Health QA Enrichment Pipeline — SPEC

## 1. Purpose and Goal

This project creates a single, source-faithful inventory of consumer-health questions drawn from a defined set of public datasets, then enriches each question with a structured annotation generated by Anthropic Claude Haiku 4.5 on Amazon Bedrock.

The final deliverable is not a medical-answering system. It is an annotation pipeline for question analysis. The two core outputs are:

1. a canonical question inventory in JSONL; and  
2. a second JSONL where each question has:
   - a difficulty rating from 1 to 5,
   - 3–7 high-information-gain follow-up questions,
   - a short explanation for why each follow-up is useful,
   - a weight from 1 to 5 for the expected information gain.

The design goal is a source-faithful question inventory with lightweight source traceability where practical. The required outputs stay minimal. Provenance sidecars are useful for auditability, but they are optional rather than a hard requirement for v1.

### Non-goals

This spec does **not** define:

- coding standards,
- repository layout beyond the core process artifacts,
- deployment or orchestration details,
- model-evaluation methodology,
- answer-generation or clinical-advice behavior.

## 2. Datasets in Scope

The project scope is limited to the following consumer-health QA datasets:

1. MedQuAD  
2. LiveQA-Med  
3. MEDIQA-QA  
4. MedicationQA  
5. MASH-QA  
6. MedRedQA  
7. HealthSearchQA  

## 2.1 Canonical dataset names

Use the following dataset strings throughout the pipeline:

| Canonical name | Dataset |
|---|---|
| `medquad` | MedQuAD |
| `liveqa_med` | LiveQA Medical / TREC 2017 |
| `mediqa_qa` | MEDIQA 2019 Task 3 QA |
| `medication_qa` | MedicationQA |
| `mash_qa` | MASH-QA |
| `medredqa` | MedRedQA |
| `health_search_qa` | HealthSearchQA |

These names are the stable dataset identifiers used in:

- the pass-1 unified JSONL,
- the internal join key `dataset + ":" + id`,
- the pass-2 enriched JSONL,
- any reconciliation or audit tooling.

## 2.1.1 Canonical id rule

`id` must be the bare source identifier within its dataset.

Do **not** prefix the dataset name inside `id`.

Examples:

- good: `0000805-1`
- good: `train_000001`
- good: `42`
- do not use: `medquad:0000805-1`
- do not use: `medredqa:train_000001`

Dataset namespacing is already provided by:

- the separate `dataset` field in JSONL; and
- the internal join key `dataset + ":" + id`.

## 2.2 Text-preservation policy

The project should keep the question text as close as possible to the original dataset representation.

Allowed normalization:

- decode XML / HTML entities (for example `&apos;` → `'`);
- normalize line endings to `\n`;
- trim outer whitespace;
- join multi-field question parts with `\n\n` when the dataset stores one user question across multiple fields.

Do **not**:

- rewrite,
- summarize,
- spell-correct,
- expand abbreviations,
- standardize capitalization,
- remove typos.

The point of the project is to annotate the question as it appears in the source dataset.

## 2.3 Dataset source, download, question field, and record-connection rules

### 2.3.1 `medquad`

**Source / origin**  
Official MedQuAD repository.

**Primary download**  
Clone or download the official GitHub repository.

**Practical source**  
Use the official XML files in the repository as the canonical source of truth.

**Question location**  
Each XML file contains a `<Document>` with one or more `<QAPair>` elements. The question text is the text content of `<Question>`. The stable question id is the `qid` attribute on `<Question>`.

**Canonical extraction**

- `dataset`: `medquad`
- `id`: value of `Question@qid`
- `question`: text content of `<Question>`

**How to reconnect to the original record**

- Locate the XML file containing `<Question qid="...">`.
- The parent `<Document id="...">` and repository file path provide document-level provenance.
- The exact source question is the matching `<Question>` node under its parent `<QAPair>`.

**Notes**

- Keep one output record per `<Question>`.
- Do not collapse multiple `<QAPair>` elements from the same XML file.

### 2.3.2 `liveqa_med`

**Source / origin**  
Official TREC 2017 LiveQA Medical dataset repository. A flattened Hugging Face mirror is available and is useful for quick inspection and loading.

**Primary download**  
Use the official GitHub repository.  
The Hugging Face mirror is acceptable for flattened access if the official XML is inconvenient.

**Recommended unit of analysis**  
Use the **original consumer question**, not answer candidates and not derived answer-ranking rows.

**Question location (official annotation structure)**  
The official question annotation structure includes:

- `qid` on `<NLM-QUESTION>`
- `<Original-Question qfile="">`
  - `<SUBJECT>`
  - `<MESSAGE>`
- `<NIST-PARAPHRASE>`
- `<NLM-Summary>`

**Canonical extraction**

- `dataset`: `liveqa_med`
- `id`: `qid` (or `QUESTION_ID` in flattened mirrors)
- `question`:
  1. if both `SUBJECT` and `MESSAGE` are non-empty: `SUBJECT + "\n\n" + MESSAGE`
  2. else use the non-empty one
  3. fallback: `NLM-Summary`

**Why this rule**  
The project is trying to preserve the raw consumer phrasing wherever possible. `NLM-Summary` is useful as a fallback, but it is not the preferred canonical question if the original subject/message fields are present.

**How to reconnect to the original record**

- Locate the `<NLM-QUESTION qid="...">` node in the official XML or the row with matching `QUESTION_ID` in a flattened mirror.
- Use `qfile` / `ORIGINAL_QUESTION_FILE` to reconnect to the raw original question file where available.

**Notes**

- Do not use answer candidates as separate questions.
- Do not substitute `NIST-PARAPHRASE` as the canonical question unless this becomes a separate experimental variant.

### 2.3.3 `mediqa_qa`

**Source / origin**  
MEDIQA 2019 Task 3 (QA). Official XML files are in the MEDIQA2019 repository. A BigBio / Hugging Face mirror exposes a convenient per-question view.

**Primary download**  
Use the official MEDIQA 2019 Task 3 QA files or the BigBio question-level mirror.

**Recommended unit of analysis**  
Use one record per **question**, not one record per answer candidate.

**Question location**

- Official XML: `<Question QID="..."><QuestionText>...</QuestionText> ... </Question>`
- BigBio mirror:
  - `question_id`
  - `question`

**Canonical extraction**

- `dataset`: `mediqa_qa`
- `id`: `QID` / `question_id`
- `question`: `QuestionText` / `question`

**How to reconnect to the original record**

- Locate the `<Question QID="...">` node in the official XML split; or
- locate the row with matching `question_id` in the BigBio mirror.

**Notes**

- MEDIQA-QA contains candidate answers and answer-ranking labels. Those are not part of pass 1.
- Preserve the question once, at question level.

### 2.3.4 `medication_qa`

**Source / origin**  
Official MedicationQA repository, distributed as an Excel workbook. A Hugging Face mirror exposes the fields in tabular form.

**Primary download**  
Use the official workbook as canonical.  
The Hugging Face mirror is acceptable for quick loading and schema inspection.

**Question location**

- Workbook column: `Question`
- Related fields commonly exposed in mirrors:
  - `Focus (Drug)`
  - `Question Type`
  - `Answer`
  - `Section Title`
  - `URL`

**Canonical extraction**

- `dataset`: `medication_qa`
- `id`: bare worksheet row number from the frozen canonical workbook, stored as a string, e.g. `"2"`
- `question`: value from the `Question` column

**Why row ids are required**  
The public tabular view exposes the question text and related metadata, but not a stable explicit record id that should be relied on across tools. The project therefore uses the workbook row number because it is deterministic and directly locatable in the canonical file.

**How to reconnect to the original record**

- Open the canonical workbook.
- Use the stored worksheet row number to locate the exact row.
- The workbook row is the record of truth.

**Notes**

- Freeze the workbook version used for ingestion.
- Freeze the worksheet name used for ingestion.
- Do not renumber rows from a resorted workbook or differently sorted export.

### 2.3.5 `mash_qa`

**Source / origin**  
Official MASH-QA repository and linked dataset archive. A Hugging Face JSON mirror exposes a simple `id` / `question` / `answers` structure.

**Primary download**  
Use the official dataset archive linked from the repository when available.  
The Hugging Face JSON mirror is acceptable for direct ingestion if it matches the intended question-level representation.

**Question location**

- `id`
- `question`

**Canonical extraction**

- `dataset`: `mash_qa`
- `id`: `id`
- `question`: `question`

**How to reconnect to the original record**

- Match on `id` in the canonical source file.

**Notes**

- MASH-QA is already close to the desired pass-1 unit.
- Do not transform answer text into additional question records.

### 2.3.6 `medredqa`

**Source / origin**  
Official paper plus CSIRO-hosted data collection. A practical Hugging Face mirror exposes the table with `Title`, `Body`, `Response`, and related metadata.

**Primary download**

- Preferred provenance source: official paper / official data collection.
- Practical programmatic source: Hugging Face mirror.

**Question location**
The user question is split across:

- `Title`
- `Body`

The mirror also exposes:

- `Unnamed: 0` (index-like column)
- `Response`
- `Response Score`
- `Occupation`
- `PMCID(s)`

**Canonical extraction**

- `dataset`: `medredqa`
- `id`:
  - use `Unnamed: 0` if present and stable in the chosen source;
  - otherwise generate a deterministic bare synthetic id from split + row number, e.g. `train_000001`
- `question`:
  - if both `Title` and `Body` are non-empty: `Title + "\n\n" + Body`
  - else use the non-empty field

**Why this rule**  
The title and body together are the actual consumer prompt. Using only one field often drops critical context.

**How to reconnect to the original record**

- If using the mirror and `Unnamed: 0` is retained, locate the row by that value.
- If using synthetic ids, locate the record by split and row number in the frozen canonical export.

**Notes**

- Do not strip demographic or symptom details from the body.
- Do not try to compress the title/body into a single rewritten sentence.

### 2.3.7 `health_search_qa`

**Source / origin**  
HealthSearchQA from the Med-PaLM / MultiMedQA work. An official supplemental XLSX is published with the Nature article, and a Hugging Face mirror exposes `id` and `question`.

**Primary download**

- Preferred canonical source: the official supplemental XLSX.
- Practical mirror: Hugging Face.

**Question location**

- `id`
- `question`

**Canonical extraction**

- `dataset`: `health_search_qa`
- `id`: bare worksheet row number from the frozen canonical XLSX, stored as a string, e.g. `"2"`
- `question`: question text from the chosen frozen source row

**How to reconnect to the original record**

- Locate the row by worksheet row number in the chosen frozen XLSX.
- If a mirror is used for loading convenience, map each mirror row back to the canonical workbook row before writing pass 1.

**Notes**

- This dataset is question-only. There is no answer field to ingest for pass 1.
- If a mirror `id` disagrees with workbook row numbering, the workbook row number wins for the canonical `id`.

## 2.4 Minimal record contract and optional provenance

The core pass-1 JSONL contains only:

- `dataset`
- `id`
- `question`

These three fields are the only required fields in the core pass-1 JSONL.

### Recommended but optional sidecar manifest

A sidecar manifest is recommended, even though it is not part of the required core JSONL.

Suggested manifest schema:

```json
{
  "dataset": "medquad",
  "id": "0000805-1",
  "split": "train",
  "source_file": "4_MPlus_Health_Topics_QA/0000805.xml",
  "source_locator": "Document[id=0000805] > QAPair[pid=1] > Question[qid=0000805-1]",
  "source_url": "https://github.com/abachaa/MedQuAD"
}
```

The manifest is useful for:

- manual audit,
- debugging,
- regeneration,
- explaining where any generated annotation came from.

It is not required for v1 acceptance.

## 2.5 Duplicate policy

Do **not** deduplicate across datasets in the canonical pass-1 JSONL.

Reason:

- some datasets intentionally reuse or reformulate questions from other datasets;
- provenance matters more than uniqueness;
- the same consumer question appearing in two datasets is analytically meaningful.

Optional later work:

- add a separate duplicate-analysis artifact based on normalized question fingerprints.

## 3. Two-pass process

## 3.1 Pass 1 — Build the unified question JSONL

### Goal

Produce a single JSONL file containing every in-scope question record in canonical form.

### Required output

One JSON object per line:

```json
{"dataset":"medquad","id":"0000805-1","question":"Do you have information about Seniors' Health"}
```

### Required fields

- `dataset`: canonical dataset name from section 2.1
- `id`: dataset-stable id as defined in section 2.3
- `question`: source-faithful question text as defined in section 2.3 and normalized only under section 2.2

### File name

Recommended canonical output:

- `all_consumer_health_questions.jsonl`

### Inclusion rules

- Include all question records from all in-scope datasets using the dataset-specific unit of analysis defined above.
- Include every public question-bearing split or partition available for every in-scope dataset in the chosen frozen source.
- If a dataset has no split concept, ingest the full public question-bearing dataset.
- Preserve one line per source question record.

### Exclusion rules

Do **not** include:

- answer candidates,
- long answers,
- labels,
- retrieval contexts,
- supporting documents,
- PMCIDs,
- focus/type annotations,
- summaries as stand-alone records,
- evaluation metadata.

Those may exist in source datasets, but they are not part of the pass-1 unified question inventory.

### Ordering

Use a deterministic order so the file is reproducible. Recommended order:

1. `medquad`
2. `liveqa_med`
3. `mediqa_qa`
4. `medication_qa`
5. `mash_qa`
6. `medredqa`
7. `health_search_qa`

Within each dataset, preserve source order if practical. If source order is not stable, sort by canonical `id`.

### Validation checks for pass 1

Every line must satisfy:

- valid JSON object;
- exactly the required fields unless the team explicitly chooses to add extra fields in a separate experimental variant;
- non-empty `dataset`, `id`, and `question`;
- `dataset` must be one of the canonical names;
- `id` must be a string;
- `question` must be a string;
- no duplicate `(dataset, id)` pairs.

### Pass-1 acceptance criteria

Pass 1 is complete when:

- the unified JSONL parses line by line,
- every line satisfies the validation checks.

If optional provenance sidecars are produced, every `(dataset, id)` should reconnect to one and only one source record under the rules in section 2.3.

## 3.2 Pass 2 — Generate structured annotations with Anthropic Claude Haiku 4.5 on Amazon Bedrock

### Goal

For each record in the pass-1 JSONL, generate a structured JSON object containing:

- the original user question,
- dataset name,
- source id,
- a difficulty level from 1 to 5,
- 3–7 expert follow-up questions,
- a short rationale for each follow-up,
- an information-gain weight from 1 to 5 for each follow-up.

### Recommended Bedrock model choice

For reproducibility, prefer the pinned Bedrock model id:

- `anthropic.claude-haiku-4-5-20251001-v1:0`

For one-off smoke tests through the Anthropic Bedrock SDK wrapper, the equivalent resolved Bedrock runtime id is:

- `us.anthropic.claude-haiku-4-5-20251001-v1:0`

### Recommended API surface

Recommended process:

1. dry-run a single request with a one-off Bedrock Claude call;
2. run the corpus with repeated one-off Bedrock Claude calls through the existing helper.

Why:

- it reuses the existing tested runtime path;
- it avoids extra S3 and IAM setup;
- it supports simple resume logic and cheap smoke tests with `--limit 1`.

### Required input to pass 2

Each pass-1 line is the model input unit:

```json
{"dataset":"health_search_qa","id":"1","question":"How serious is atrial fibrillation?"}
```

### Required output schema per record

The generated object must preserve traceability and must follow this shape:

```json
{
  "id": "1",
  "dataset": "health_search_qa",
  "user": "How serious is atrial fibrillation?",
  "difficulty_level": 2,
  "questions": [
    {
      "question": "Is this about you or someone else right now, and are there any red-flag symptoms such as chest pain, severe shortness of breath, fainting, confusion, weakness or numbness, or trouble speaking or seeing?",
      "thinking": "This separates a possible emergency from a routine information request. Urgency changes the next step entirely, so it has the highest information gain.",
      "weight": 5
    },
    {
      "question": "Has atrial fibrillation already been diagnosed, or are you asking because of a new irregular heartbeat or new symptoms?",
      "thinking": "This distinguishes a general education question from a new possible diagnosis. The answer changes what context is missing and what follow-up matters most.",
      "weight": 4
    },
    {
      "question": "What other health conditions, medications, or blood thinners are involved?",
      "thinking": "Comorbidities and current treatment strongly affect risk framing and what information would be most useful next.",
      "weight": 5
    }
  ]
}
```

### Field rules

#### Top level

- `id`: must exactly equal the pass-1 `id`
- `dataset`: must exactly equal the pass-1 `dataset`
- `user`: must exactly equal the pass-1 `question`
- `difficulty_level`: integer in `[1, 5]`
- `questions`: array length in `[3, 7]`

#### Follow-up question objects

Each follow-up object must have:

- `question`: non-empty string
- `thinking`: non-empty string explaining the information gain
- `weight`: integer in `[1, 5]`

### Semantics

#### Difficulty level

Difficulty is the difficulty of answering the question **well from the raw question alone**, not:

- the severity of the medical issue,
- the emotional intensity,
- the urgency,
- the likelihood of harm.

#### Follow-up questions

Each follow-up question should:

- materially reduce ambiguity,
- help identify urgency, context, or hidden intent,
- be non-redundant,
- be phrased naturally,
- maximize information gain.

#### Weight

`weight` is the expected information gain:

- `1`: low information gain
- `5`: very high information gain

### Prompt contract

The model prompt should instruct Claude to:

- return **JSON only**,
- return exactly one object,
- preserve `dataset`, `id`, and `user` exactly,
- generate 3–7 follow-up questions,
- keep `difficulty_level` and `weight` in the integer range 1–5,
- make follow-up questions non-redundant and targeted at distinct information gaps,
- avoid markdown,
- avoid code fences,
- avoid extra commentary before or after the JSON.

### Recommended request structure

At a high level, each request should include:

- a top-level system instruction that defines the annotation task and schema;
- a user message containing:
  - dataset,
  - id,
  - question text,
  - the exact output schema.

### Execution strategy

Use the pass-1 JSONL as the work queue and process records one by one.

For each input row:

- derive `logical_key = dataset + ":" + id`;
- call the existing Bedrock helper with the pinned Haiku 4.5 model;
- validate the JSON response immediately;
- if invalid, repair it in place before moving on;
- append valid outputs to the enriched JSONL;
- append permanent failures to a failure log.

Example:

- `health_search_qa:1`
- `medquad:0000805-1`

### Why the join key matters

`logical_key` is the stable join key between:

- the pass-1 input row,
- repair attempts,
- failure logs,
- the pass-2 output row.

### Recommended pass-2 file outputs

Required:

- `all_consumer_health_questions_enriched.jsonl`

Recommended operational artifacts:

- `run_manifest.json`
- `annotation_failures.jsonl`
- `repair_attempts.jsonl`

## 3.2.1 Validation and repair loop

Every model output must be validated before being accepted into the final enriched JSONL.

### Validation checks

Validation must include both structural checks and semantic checks.

#### Structural validation checks

For each response:

1. parse as JSON;
2. confirm top-level shape;
3. confirm `id`, `dataset`, and `user` exactly match input;
4. confirm `difficulty_level` is an integer 1–5;
5. confirm there are 3–7 follow-up questions;
6. confirm every follow-up has:
   - `question`,
   - `thinking`,
   - `weight`;
7. confirm every `weight` is an integer 1–5;
8. confirm strings are non-empty;
9. confirm no markdown fences or wrapper text leaked into the output.

#### Semantic validation checks

For each response:
10. confirm the normalized follow-up question strings are all distinct;
11. confirm no follow-up is a trivial restatement of the user question;
12. confirm each `thinking` field explains a concrete ambiguity or decision-relevant information gap, not generic filler;
13. confirm the follow-up set covers at least 3 distinct information axes when the raw question allows it, chosen from:

- urgency / red flags,
- chronology / severity,
- patient context,
- prior evaluation / treatment,
- relevant history / medications,
- user intent / goals;

14. reject outputs with near-duplicate follow-ups even if they pass structural schema checks.

### Recommended default semantic validator

Use a two-stage semantic gate in v1:

1. deterministic screening:
   - normalize case, whitespace, and punctuation;
   - reject exact duplicates;
   - reject obvious near-duplicates using a conservative lexical similarity threshold;
2. model-assisted review:
   - use a lightweight reviewer prompt to check non-redundancy and information gain;
   - if the reviewer rejects the set, send the record through repair.

A reasonable default for v1 is:

- lexical near-duplicate threshold: `SequenceMatcher >= 0.88`;
- minimum semantic breadth: at least 3 distinct information axes when the source question allows it;
- repair budget: 2 repair attempts after the initial invalid output;
- backoff: 2 seconds before repair attempt 1, then 8 seconds before repair attempt 2.

### Repair policy

If validation fails:

1. send a repair prompt containing:
   - the original input,
   - the invalid model output,
   - the validation error(s),
   - the instruction to return corrected JSON only;
2. retry up to **2 repair attempts** after the initial failed output, with exponential backoff of **2 seconds** and **8 seconds** between repair calls;
3. if still invalid, write the record to a failure file and do not silently drop it.

### Acceptance rule

Only validated JSON objects are allowed into the final enriched JSONL.

## 3.2.2 Resume and idempotency

Pass 2 must be resumable.

Recommended behavior:

- if an output line already exists for `(dataset, id)` and passes validation, skip regeneration unless a full rerun is explicitly requested;
- preserve a run manifest containing:
  - model snapshot,
  - prompt version,
  - run timestamp,
  - input file path or checksum,
  - processed / skipped / failed counts.

This ensures reproducibility and clean reruns.

## 4. Canonical process summary

### Step A — Freeze sources

Freeze the exact source versions / files used for each dataset.

### Step B — Extract canonical questions

Apply the dataset-specific rules in section 2.3.

### Step C — Write pass-1 JSONL

Write one line per question:

```json
{"dataset":"...","id":"...","question":"..."}
```

### Step D — Validate pass 1

Check schema and uniqueness. If optional provenance sidecars are produced, verify reconnectability.

### Step E — Dry-run Bedrock on one record

Use a one-off Bedrock Claude call with the pinned Haiku 4.5 model id and validate the response shape.

### Step F — Run pass 2 at scale

Iterate through the pass-1 JSONL, call the existing Bedrock helper one record at a time, and preserve `logical_key = dataset:id` in operational logs.

### Step G — Validate and repair results

Accept only structurally valid and semantically valid JSON objects.

### Step H — Write the final enriched JSONL

Write one validated enriched object per input question.

## 5. Final deliverables

### Required deliverables

1. `all_consumer_health_questions.jsonl`  
   Canonical pass-1 question inventory.

2. `all_consumer_health_questions_enriched.jsonl`

2. run manifest  
3. failure log  
4. repair log
